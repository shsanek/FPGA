# RAM_CONTROLLER

Мост между кэш-контроллером (`MEMORY_CONTROLLER`) и интерфейсом Xilinx MIG7 (DDR).
Работает на двух тактовых доменах: `clk` (системный, 100 МГц) и `mig_ui_clk` (MIG, 125 МГц).
Принимает одиночные запросы чтения/записи 128-битными чанками и транслирует их в MIG-транзакции.

---

## Параметры

| Параметр       | По умолчанию | Описание                              |
|----------------|-------------|---------------------------------------|
| `CHUNK_PART`   | 128         | Ширина данных в битах                 |
| `ADDRESS_SIZE` | 28          | Ширина адреса в битах                 |
| `CHUNK_COUNT`  | 4           | Не используется внутри модуля         |

## Интерфейс

### Системный домен (`clk`)

| Сигнал           | Нап. | Описание                                                          |
|------------------|------|-------------------------------------------------------------------|
| `clk`            | in   | Системный тактовый сигнал                                        |
| `controller_ready` | out | Контроллер готов принять новый запрос                           |
| `error[3:0]`     | out  | Флаг ошибки (выставляется, если `WATING` получил пустой запрос) |
| `write_trigger`  | in   | Импульс 1 такт — начать запись                                   |
| `write_value[CHUNK_PART-1:0]` | in | Данные для записи                              |
| `write_address[ADDRESS_SIZE-1:0]` | in | Адрес записи                              |
| `read_trigger`   | in   | Импульс 1 такт — начать чтение                                   |
| `read_address[ADDRESS_SIZE-1:0]` | in | Адрес чтения                               |
| `read_value[CHUNK_PART-1:0]` | out | Прочитанные данные                             |
| `read_value_ready` | out | Импульс 1 такт — данные на `read_value` валидны               |
| `led0[2:0]`      | out  | Диагностические светодиоды                                       |

### MIG-домен (`mig_ui_clk`)

Стандартный интерфейс Xilinx MIG7 app-port:

| Сигнал                 | Нап. | Описание                            |
|------------------------|------|-------------------------------------|
| `mig_app_addr`         | out  | Адрес команды                       |
| `mig_app_cmd[2:0]`     | out  | Команда: `000`=write, `001`=read    |
| `mig_app_en`           | out  | Строб команды                       |
| `mig_app_wdf_data`     | out  | Данные записи                       |
| `mig_app_wdf_wren`     | out  | Строб записи данных                 |
| `mig_app_wdf_end`      | out  | Признак последнего слова записи     |
| `mig_app_wdf_mask`     | out  | Маска байт (всегда `0` — без маски) |
| `mig_app_wdf_rdy`      | in   | MIG готов принять данные записи     |
| `mig_app_rd_data`      | in   | Прочитанные данные из DDR           |
| `mig_app_rd_data_valid`| in   | Данные чтения валидны               |
| `mig_app_rd_data_end`  | in   | Последнее слово ответа чтения       |
| `mig_app_rdy`          | in   | MIG готов принять команду           |
| `mig_ui_clk`           | in   | Тактовый сигнал MIG                 |
| `mig_init_calib_complete` | in | DDR-калибровка завершена          |

## Принцип работы

### Инициализация

После сброса контроллер ждёт, пока `mig_init_calib_complete` станет `1`. До этого момента `controller_ready = 0`. Когда калибровка завершается, запускается CDC-handshake (см. ниже) для передачи управления в домен `clk`.

### Обработка запроса

Для каждой операции (чтение или запись) контроллер проходит один полный CDC-цикл: `clk`-домен захватывает сигналы, передаёт управление `mig_ui_clk`-домену, MIG-домен исполняет транзакцию, затем возвращает управление обратно.

**Шаги для чтения (`read_trigger = 1`):**

1. `clk`-домен фиксирует `read_address` и `read_trigger` во внутренних регистрах, опускает `controller_ready`, переходит в `WILL_STOP_CONTROLL`.
2. CDC-handshake передаёт управление `mig_ui_clk`-домену.
3. `mig_ui_clk`-домен (`WATING`): ждёт `mig_app_rdy`, выставляет `mig_app_en`, `mig_app_cmd=001`, переходит в `READ`.
4. `READ`: ждёт `mig_app_rd_data_valid`, сохраняет данные в `internal_output_value`, переходит в `WATING` и инициирует возврат управления.
5. CDC-handshake возвращает управление `clk`-домену.
6. `clk`-домен выставляет `read_value_ready = 1` на один такт, снова поднимает `controller_ready`.

**Шаги для записи (`write_trigger = 1`):**

Аналогично чтению, но в шаге 3 `mig_ui_clk`-домен выставляет `mig_app_cmd=000`, `mig_app_wdf_wren`, `mig_app_wdf_end`. Ждёт, пока `mig_app_en` и `mig_app_wdf_wren` оба обнулятся (подтверждение MIG), затем инициирует возврат.

**Одновременные запрос записи + чтения (`write_trigger` и `read_trigger` в одном такте):**

Контроллер выполняет сначала **запись**, затем **чтение** за один CDC-цикл без возврата в `clk`-домен между ними. Флаг `skip_write` предотвращает повторное выполнение записи при повторном заходе в `WATING`.

## CDC-Handshake (двухтактовый четырёхфазный протокол)

Обмен управлением между доменами реализован через четыре состояния `SYNC_CONTROLLER_STATE`:

```
ACTIVE_CONTROLL → WILL_STOP_CONTROLL → NOT_CONTOLL → WILL_START_CONTROLL → ACTIVE_CONTROLL
```

Каждый домен имеет собственный регистр состояния (`controll_clk_state`, `controll_ui_clk_state`). Переход происходит только тогда, когда оба домена находятся в ожидаемых парных состояниях — это гарантирует корректную CDC без метастабильности для данного простого протокола.

| Шаг | `controll_clk_state`    | `controll_ui_clk_state`  | Что происходит                        |
|-----|-------------------------|--------------------------|---------------------------------------|
| 1   | `ACTIVE_CONTROLL`       | любое                    | `clk`-домен работает, виден запрос    |
| 2   | `WILL_STOP_CONTROLL` ←  | `WILL_START_CONTROLL`    | `clk`: подтверждает остановку         |
| 3   | `NOT_CONTROLL`          | `WILL_STOP_CONTROLL` ←   | `ui_clk`: принимает управление        |
| 4   | `WILL_START_CONTROLL` ← | `NOT_CONTROLL`           | `clk`: готовится принять управление   |
| 5   | `ACTIVE_CONTROLL` ←     | `WILL_START_CONTROLL`    | `ui_clk`: отдаёт управление           |

---

## Состояния MIG-домена (`RAM_CONTROLLER_STATE`)

```
INIT → WATING ⇄ READ
              ⇄ WRITE
```

| Состояние | Описание                                                                         |
|-----------|----------------------------------------------------------------------------------|
| `INIT`    | Ожидание `mig_init_calib_complete`. После — CDC-handshake → `clk` получает управление. |
| `WATING`  | Ожидание `mig_app_rdy`. Dispatch: write → `WRITE`, read → `READ`, ни того ни другого → `error`. |
| `READ`    | Ждёт `mig_app_rd_data_valid`, сохраняет данные, инициирует возврат управления.  |
| `WRITE`   | Ждёт подтверждения от MIG (`!en && !wdf_wren`). Если был также `read_trigger` — переходит обратно в `WATING` с `skip_write=1` для выполнения чтения. |

## Диагностические светодиоды (`led0`)

| Бит     | Сигнал                                                    |
|---------|-----------------------------------------------------------|
| `led0[0]` | `!mig_init_calib_complete` — горит до калибровки DDR    |
| `led0[1]` | `mig_init_calib_complete` — горит после калибровки      |
| `led0[2]` | `clk`-домен не в `ACTIVE_CONTROLL` — идёт операция      |

## Ограничения и особенности

- **Один запрос за раз.** Пока `controller_ready = 0`, новые `write_trigger`/`read_trigger` игнорируются.
- **Маска всегда 0.** `mig_app_wdf_mask` жёстко прошит в `0` — записываются все 128 бит.
- **`controller_ready = 0` при сбросе.** Внешний модуль должен дождаться `controller_ready = 1` после включения (после калибровки MIG и завершения начального CDC-цикла).
- **Ошибка при пустом запросе.** Если `mig_ui_clk`-домен вошёл в `WATING` но ни `internal_write_trigger`, ни `internal_read_trigger` не выставлены — поднимается `error`. Этого не должно происходить в нормальной работе.
